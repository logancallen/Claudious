### 6. Operator Pattern with --allowedTools Scoping
**Source:** https://www.mindstudio.ai/blog/claude-code-agentic-workflow-patterns
**Finding-ID:** 2026-04-14-operator-pattern-allowedtools
**Category:** PATTERN
**What it does:** Operator pattern (central orchestrator + specialized subagents) becomes dramatically more reliable when the orchestrator uses `--allowedTools` to restrict what each subagent can do. Prevents subagent scope creep and makes results deterministic.
**Implementation:**
1. Define orchestrator with narrow tool scope: `claude --allowedTools Task,Bash`
2. Each spawned subagent gets its own scope: frontend agent `--allowedTools Edit,Read,Glob` (no Bash); DB agent `--allowedTools Bash,Read`
3. Encode scopes in subagent YAML frontmatter under `.claude/agents/*.md`
4. Orchestrator's job: decompose task, dispatch to scoped agents, merge results
5. Pattern prevents e.g. a frontend agent from accidentally running migrations
**Risk:** TEST-FIRST (requires subagent refactor)

### 9. MCP Circuit Breaker — 50%/30s Reliability Pattern
**Source:** https://dextralabs.com/blog/claude-code-mcp-enterprise-ai-integrations/
**Finding-ID:** 2026-04-14-mcp-circuit-breaker
**Category:** PATTERN
**What it does:** Specific circuit breaker config for MCP calls: open after 50% failure rate in a 10-second sliding window, half-open after 30 seconds to test recovery. Combined with exponential backoff + full jitter, 3-5 max retries. Prevents cascading failures when an MCP server goes down.
**Implementation:**
1. Wrap all MCP tool invocations in a retry-with-jitter policy
2. Exponential backoff: `delay = random(0, base * 2^attempt)` where base=100ms
3. Max 3-5 attempts per invocation
4. Circuit breaker thresholds: 50% failure over 10s → open; 30s cooldown → half-open test
5. Log circuit state transitions to surface degraded MCP servers
6. For Logan: useful if Supabase MCP starts flapping — prevents wasting tokens on dead connections
**Risk:** TEST-FIRST (requires wrapper around MCP client calls)

3. Share the spec. Ask: "Review this spec and propose an implementation plan. Do not write code."
4. Iterate on the plan — challenge assumptions, require rationale
5. Only after plan approval, exit plan mode and execute
